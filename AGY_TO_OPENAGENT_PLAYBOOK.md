## 1. The Resonant Planning & Delegation Protocol (BKM-030.1)

To transition tasks between AGY (Gemini-driven, high-fidelity co-pilot) and OpenAgent (Groq/Ollama-driven, autonomous swarm) without state thrashing or intent loss:

```
  ┌────────────────────────────────────────────────────────┐
  │                 1. Planning (AGY / Gemini)             │
  │  Draft Master Sprint Plan (How, Why, Verification)    │
  └───────────────────────────┬────────────────────────────┘
                              │
                              ▼
  ┌────────────────────────────────────────────────────────┐
  │               2. Handover (AGY -> OpenAgent)            │
  │  Translate task segments into OpenCode backlogs/tasks  │
  └───────────────────────────┬────────────────────────────┘
                              │
                              ▼
  ┌────────────────────────────────────────────────────────┐
  │             3. Execution (OpenAgent Swarm)             │
  │  Run parallel coding and reviews on Groq / local 4090   │
  └───────────────────────────┬────────────────────────────┘
                              │
                              ▼
  ┌────────────────────────────────────────────────────────┐
  │                 4. Review (AGY / Gemini)               │
  │  Verify commits, run diagnostics, merge to timeline    │
  └────────────────────────────────────────────────────────┘
```

### Protocol Steps

1.  **Orchestrator Planning Phase (AGY):**
    *   Write the Master Sprint Plan (`SPRINT_PLAN_SPR_XX_X.md`). Every task must detail the **Why** (strategy), the **How** (files/logic), and the **Verification** (specific tests).
    *   *Rule:* AGY (Gemini) maintains the Master Plan because it has the context capacity to prevent "Loss of Intent" (BKM-020).
    *   *Note on `/conductor`:* The `/conductor` directory is deprecated for active planning; the sprint markdown files remain the sole source of truth for planning and tracking.
2.  **Card/Task Synthesis (AGY -> OpenAgent):**
    *   For tasks delegated to OpenAgent, AGY adds tasks directly to the workspace task list or generates a local workspace task.
    *   Include the verbatim **Verification** block from the Master Plan.
3.  **Swarm Play (OpenAgent):**
    *   Trigger OpenAgent to execute the task using the balanced model mappings (Ollama for main coding, Groq/DeepSeek for parallel tests/critiques).
    *   OpenAgent executes the code changes, runs the verification tests, and commits the results locally.
4.  **Forensic Gatekeeper (AGY):**
    *   AGY inspects the git diffs generated by OpenAgent.
    *   AGY runs the system-wide diagnostic check (e.g., `/heartbeat`, local python test scripts) to certify the change.
    *   Once validated, AGY checks off the task in the Master Sprint Plan.

## 2. Model-Agnostic Intent Anchoring (BKM-015)

When Google API tokens are exhausted, the interface to the code must transition seamlessly. **Intent Anchors** act as the translation layer.

### How Intent Anchors Bridge the Gap
*   **The Problem:** Gemini models and Llama/DeepSeek models parse natural language prompts differently. If tool execution is gated by specific prompt phrasings, switching models causes immediate failure.
*   **The Solution:** All tools, file targets, and architectural rules (from `FeatureTracker.md`) are mapped to semantic keys inside `config/intent_anchors.json`.
*   **Routing Logic:**
    1.  The client (AGY or OpenAgent) parses the active task or user prompt.
    2.  Instead of using a hardcoded `if/else` keyword check, it queries the local vector store (`behavioral_dna` collection) or parses the regex intent map.
    3.  This translates the intent (e.g., `"Update RAPL telemetry"`) to the correct code module and tool parameter (e.g., `[FEAT-098]`, `/monitor/rapl_sim/app.py`).
    4.  Because the anchors are model-agnostic, the task executes identically whether driven by Gemini 2.5/3.5 or Omnicoder-9B.

## 6. Vector Grounding & Synchronization via ChromaDB

To optimize prompt sizes and increase query speed, the system can utilize a local vector store (ChromaDB) to retrieve relevant rules and technical facts.

### 6.1 Markdown as the Single Source of Truth
*   **The Law:** The plain-text files `FeatureTracker.md` and `Protocols.md` remain the absolute, single source of truth. They are version-controlled in Git and edited directly by humans.
*   **The Runtime vs. Development Distinction:** 
    *   *Production Runtime:* `FEAT-050` (Vibe Check on Save) is active *only* on the target Linux RTX 2080 Ti running the `acme_lab` resident service.
    *   *Development Workspace:* We do not have a background daemon watcher running in the development workspace. Running a continuous watcher process during development is resource-heavy and unnecessary.
*   **Development-Side Sync Hook Options:**
    1.  **Git Pre-Commit Hook (Recommended):** A git pre-commit hook (`.git/hooks/pre-commit`) that automatically executes `python3 sync_chroma_dna.py` before committing changes. Since Git is our system of record, this guarantees the vector database is updated on every save/commit cycle without background process overhead.
    2.  **Safe-Scalpel Integration:** Integrate the parser call directly into `atomic_patcher.py` so that any AI-driven modifications automatically trigger a DB sync.

### 6.2 Dual-Collection Separation (Behavior vs. Structure)
To prevent confusion between behavioral instructions (rules for the AI agent) and structural constraints (rules for the code/hardware), ChromaDB uses two isolated collections:

1.  **`behavioral_dna` Collection:**
    *   **Source:** `Protocols.md`
    *   **Content:** Agent protocols (e.g., BKM-004 QQ Protocol, BKM-011 Safe-Scalpel).
    *   **Purpose:** Guides *how* the agent behaves, logs, and interacts with the user.
2.  **`feature_dna` Collection:**
    *   **Source:** `FeatureTracker.md`
    *   **Content:** Technical specs (e.g., FEAT-030 Unity Residency, FEAT-098 RAPL-Sim).
    *   **Purpose:** Guides *what* technical constraints must be preserved in the codebase.

### 6.3 Deterministic Reverse Lookups
While vector search is semantic (fuzzy/probabilistic), the system requires deterministic mapping for code lookups. 

*   **Semantic Lookup:** Querying for `"patching files"` retrieves `BKM-011` and `BKM-012` because their semantic content matches the query concept.
*   **Metadata Lookups (Deterministic):** Each vector entry is stored with metadata tags:
    ```json
    {
      "feature_id": "FEAT-115",
      "tools": ["patch_file"],
      "status": "ACTIVE"
    }
    ```
*   **Bidirectional Mapping:**
    *   **BKM/FEAT to Tool:** Querying ChromaDB with a metadata filter on `feature_id == "FEAT-115"` returns the record containing the `"patch_file"` tool parameter.
    *   **Tool to BKM/FEAT:** Querying ChromaDB with a metadata filter on `tools CONTAINS "patch_file"` returns the specific rule containing `FEAT-115` and `BKM-012` instantly.
